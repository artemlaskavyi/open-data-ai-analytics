Мета: Опанувати базові принципи контейнеризації застосунків та навчитися запускати окремі модулі проєкту в Docker-середовищі. Закріпити навички побудови багатомодульного проєкту, роботи з Dockerfile, Docker Compose, томами, мережами та контейнерною взаємодією між сервісами.

Завдання:
Частина 1. Проєктування
Продумати контейнерну архітектуру. Визначити, які сервіси потрібні та як саме вони обмінюються даними.

Частина 2. Контейнеризація модулів
Створити Dockerfile для кожного модуля. Переконатися, що кожен сервіс запускається окремо.

Частина 3. Інтеграція через Docker Compose
Описати сервіси у compose.yaml. Налаштувати мережі, томи, порти, змінні середовища. Запустити систему цілком.

Частина 4. Демонстрація
Показати запуск через Docker, роботу БД, веб-інтерфейс та згенеровані звіти й графіки.


Розв’язання:
У межах лабораторної роботи було виконано повноцінну контейнеризацію монолітного проєкту аналізу відкритих даних про заробітну плату в Україні. 

Замість загальної теки `src/` архітектуру було розділено на 5 відокремлених сервісів: `data_load`, `data_quality_analysis`, `data_research`, `visualization` та `web`. Для кожного з них створено власний `Dockerfile` (на базі `python:3.10-slim`) із визначеним списком залежностей `requirements.txt`.

Як основне сховище для обміну даними між модулями інтегровано реляційну базу даних PostgreSQL (через офіційний образ `postgres:15-alpine`). Конфігурація доступу захищена за допомогою файлу середовища `.env`. Модуль `data_load` налаштовано на автоматичне завантаження Excel-файлу, його обробку та запис створених таблиць напряму в БД. Аналітичні модулі виконують читання з БД та генерують артефакти: модуль `data_quality_analysis` створює Markdown-звіт, `data_research` генерує JSON з висновками, а `visualization` відмальовує серію графіків. 

Для відображення фінальних результатів розроблено сервіс `web` на базі фреймворку Flask. Результати аналізу передаються до веб-інтерфейсу за допомогою спільних Docker томів (`volumes`).

Загальна оркестрація системи побудована у файлі `compose.yaml`. Запроваджено єдину ізольовану мережу `app-network`. Для дотримання правильної послідовності старту сервісів використано директиву `depends_on` (наприклад, аналітика чекає на заповнення БД модулем `data_load`, а веб-сервер чекає на закінчення роботи аналітичних модулів).

Висновок:
У результаті лабораторної роботи було успішно трансформовано монолітний набір скриптів у сучасну мікросервісну архітектуру за допомогою Docker та Docker Compose. Практично закріплено навички написання Dockerfile, оркестрації через compose.yaml, налаштування залежностей між контейнерами (`depends_on`), а також роботи зі спільними томами (`volumes`) і мережами (`networks`). Побудована система легко розгортається однією командою (`docker compose up`), що значно підвищує переносимість і надійність проєкту. Таким чином, мету лабораторної роботи повністю досягнуто.
